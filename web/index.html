§ 02 · the retriever

One SQL statement. Every architecture runs through it.

§ 02 · sql
src/sonzai/retriever_5lens.ts view on github → 
SELECT id, level, content, event_time, importance,
       entities, subject, predicate, object,
       (1 - VEC_COSINE_DISTANCE(embedding, ?)) AS vec_sim,
       (CASE WHEN LOWER(content) LIKE ? THEN 1.0 ELSE 0.0 END) AS lex_hit,
       DATEDIFF(?, event_time) AS age_days,
       (CASE WHEN ? IS NULL OR ? IS NULL
             OR (event_time BETWEEN ? AND ?) THEN 1.0 ELSE 0.3 END) AS time_match
FROM   memories
WHERE  approach      = ?
  AND  agent_id      = ?
  AND  superseded_at IS NULL                               -- supersede filter
ORDER BY (
    (0.50 * (1 - VEC_COSINE_DISTANCE(embedding, ?))         -- vector
     + 0.20 * (CASE WHEN LOWER(content) LIKE ? THEN 1.0 ELSE 0.0 END)  -- full-text
     + 0.30 * importance)                                      -- importance
  * EXP(-0.005 * GREATEST(0, DATEDIFF(?, event_time)))        -- recency decay
  * (CASE WHEN ? IS NULL OR ? IS NULL
          OR (event_time BETWEEN ? AND ?) THEN 1.0 ELSE 0.3 END)     -- temporal window
  * (CASE WHEN LOWER(subject) = ? OR LOWER(object) = ?      -- JSON entity
          THEN 1.35
          WHEN JSON_CONTAINS(entities, JSON_QUOTE(?))
          THEN 1.15 ELSE 1.0 END)
  * (CASE level WHEN 0 THEN 0.80                            -- level-weighted boost
                     WHEN 1 THEN 1.0
                     WHEN 2 THEN 1.05 ELSE 1.0 END)
) DESC
LIMIT 5;

This is the retriever. Every architecture runs through it. Only the approach predicate changes. Vector similarity, full-text LIKE, recency decay, temporal window, JSON entity boost, level-weighted boost, supersede filter — eight signals, one ORDER BY, one index plan.

§ 04 · why tidb

One row shape. Two implications.

§ 04 · tidb
§ for researchers

The comparison is a column rename, not a platform migration.

Five memory architectures, one memories table, one SQL retriever. Ablating any write-time move is a WHERE approach = ? switch. Four sync problems that would otherwise consume the bench disappear because the row shape absorbs them.

§ sync 1 · ingest fan-out

Every turn reaches all five approaches in one transaction on one table. Split across Pinecone + Qdrant + a SQL store + a graph DB, the same study is five clients, five retry policies, five failure modes to keep aligned before any turn is scored.

§ sync 2 · retrieval surface

The retrieval code path is byte-identical across all five approaches — only the approach predicate changes. Comparing Pinecone top-K against a Neo4j subgraph query also compares their APIs. The study's signal would be contaminated before it ran.

§ sync 3 · embedding parity

One 1536-dim Gemini embedding per turn, shared across all five writes via the same embedding VECTOR(1536) column. On split stacks you either embed five times (embeddings drift) or stand up a shared embedding service and plumb the vector to five clients.

§ sync 4 · supersession semantics

superseded_at IS NULL is a one-column predicate on an indexed column. The same retriever works for approaches that use supersession and approaches that leave it NULL. On a split stack, supersession logic forks per store.

The study exists because the comparison surface is trivial to build. The site you're reading is the direct consequence of that property — not of any single approach scoring well.

§ for developers

Fewer moving parts. Fewer places to break.

Agent memory in production usually means a vector store, a SQL store, a cache, a graph DB, and the sync glue that holds them in step. Every added system is a new dependency to monitor, a new schema to migrate, a new consistency model to reason about. Here the whole memory layer is one TiDB cluster.

§ ops 1 · one dependency

One connection string. One pool. One set of credentials to rotate. TiDB Cloud Serverless handles the whole memory layer — no Pinecone SLA, no Neo4j cluster, no Redis replicas, no cross-system backups to coordinate.

§ ops 2 · one schema

A new memory column — mood, valence, decay — is one ALTER TABLE. Across a split stack it's a coordinated migration: new field in the vector store's metadata, new column in Postgres, new edge property in the graph DB, and the sync code updated in lockstep.

§ ops 3 · one failure mode

When something goes wrong, the blame surface is one query plan. Across systems the failure modes compound: vector store lagged, Postgres replica behind, Redis evicted a key, graph edge never got written. Each one is a plausible cause; you get to rule all of them out.

§ ops 4 · one consistency model

The same row carries vector + full-text + JSON + structured columns + supersession state. If the write landed, all modalities see it. No dual-write races, no eventual-consistency windows where the SPO graph disagrees with the vector index about what the user said.

The claim isn't that TiDB is the fastest store at any one modality. It's that the number of systems you have to keep in step is one. Every additional system in a production memory layer is a new failure mode you shipped to your users.

Same engineering property — one row shape, one query plan — measured in two currencies. Ablations that would take a week become a column switch; production outages that would need five dashboards become a single query plan.
